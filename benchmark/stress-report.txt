  Active bottlenecks (code-level issues requiring fixes):   NONE

  All implemented optimizations (Phase 11) are functioning correctly.
  No performance thresholds were breached by framework code.

  The one "breach" reported (29 plugin failures) is an environment
  artifact — missing node_modules in the sandbox, not a code defect.

  ── Scaling ceiling observations ─────────────────────────────────────────────

  1. getAllCommands() degrades at 5 000+ commands:
     1k → 0.08ms (12k ops/s)
     5k → 1.07ms (935 ops/s)
     This is O(n) spread over primaryNames. At real-world plugin counts
     (<200 commands) this is negligible (<0.01ms).

  2. getCategoryIndex() rebuild at 10 000 commands = 0.85ms:
     This is paid only once per register/unregister cycle, then cached.
     At realistic plugin counts (50–200) rebuild is <0.1ms.

  3. Real plugin loading: 10.17ms per plugin = dominated by ESM import():
     Node.js must parse and execute each .js file on first import.
     This is inherent to ESM — not fixable at the framework level.
     The loader uses sequential walk() for correct dependency ordering.
     44 plugins at 10ms each = ~440ms cold-start. This is the expected
     startup cost for a bot with external dependency imports per plugin.

  ── Open (future consideration only) ─────────────────────────────────────────

  🟡 [LOW] registry — searchCommands() O(n×8) linear scan
     Impact: At 1 000 commands: 0.83ms. At 5 000: 1.1ms. Under threshold.
     Threshold breach: Only if plugin count exceeds ~2 000 and search
     is called on every user message (it isn't — only on .search command).
     Recommendation: Build a name-prefix trie at startup, updated on
     register/unregister. Only warranted if total plugins exceed ~2 000.
     Current status: Not needed — 43 real plugins loaded.

  🟡 [LOW] plugin-manager — getPluginsByCategory() no cache
     Impact: Used by .plugins owner command only. O(n) scan per call.
     Recommendation: Add a parallel categoryMap in PluginManager mirroring
     the registry's categoryIndex pattern.
     Current status: Acceptable — .plugins is a low-frequency owner command.

  🔵 [INFO] agent/queue — 10-minute job TTL for completed jobs
     Impact: At current load (<100 jobs/hr) negligible. If workload ever
     exceeds ~500 jobs/hr, completed job objects accumulate in _jobs Map
     for up to 10 minutes before pruning.
     Recommendation: Reduce TTL to 2–3 min if throughput exceeds 500/hr.
     Current status: Not needed at present load profile.

  🔵 [INFO] plugin loader — parallel import() for independent plugins
     Impact: Cold-start takes ~10ms per plugin (ESM import IO). Sequential
     loading of 44 plugins = ~440ms. Parallel loading could reduce this
     to ~50–80ms if plugins have no inter-dependencies.
     Recommendation: Group plugins by dependency level; load each level
     in parallel using Promise.all(). Saves ~360ms cold-start.
     Risk: Plugin ordering must be respected for dependency checking.
     Current status: Low priority. 440ms startup is acceptable.
