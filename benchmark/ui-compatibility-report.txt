BUGS FOUND AND FIXED DURING VALIDATION
  ┌─────────────────────────────────────────────────────────────────────┐
  │  BUG-01 (HIGH) — registry.js: unregisterCommand wipes unrelated     │
  │                  primaries when a conflicted alias is in cmd.aliases │
  ├─────────────────────────────────────────────────────────────────────┤
  │  Root cause:                                                         │
  │    unregisterCommand() iterated cmd.aliases and called               │
  │    commands.delete(alias) unconditionally.  When an alias had been   │
  │    SKIPPED at registration time (conflict with an existing primary), │
  │    it was never inserted into the commands Map — but the Map DID     │
  │    hold an entry for that key belonging to the primary command.      │
  │    Blind deletion therefore removed the primary's entry, making it   │
  │    invisible to getCommand() / getAllCommands() / searchCommands().   │
  │                                                                      │
  │  Impact:                                                             │
  │    Any plugin that registers an alias matching another plugin's      │
  │    primary name, then unregisters itself, silently deletes the       │
  │    victim command from the registry for the remainder of the         │
  │    process lifetime.  Help menus, category cards, and search all     │
  │    showed missing commands with no error.                            │
  │                                                                      │
  │  Fix applied (src/lib/registry.js):                                  │
  │    for (const alias of cmd.aliases) {                                │
  │      if (commands.get(alias) === cmd) commands.delete(alias);        │
  │    }                                                                 │
  │    Guard ensures only aliases that actually point to the command     │
  │    being removed are deleted.                                        │
  │                                                                      │
  │  Regression test added: Phase 1 "BUG-FIX regression" block (3 tests)│
  └─────────────────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────────────────┐
  │  BUG-02 (LOW/WA-05) — interactive.js: copyButton stores copy_code   │
  │                        as the raw JS type instead of a string        │
  ├─────────────────────────────────────────────────────────────────────┤
  │  Root cause:                                                         │
  │    copyButton(label, textToCopy) passed textToCopy directly into     │
  │    JSON.stringify without coercion.  WhatsApp's NativeFlow spec      │
  │    requires copy_code to be a JSON string; passing a number or       │
  │    boolean causes the interactive message to be silently dropped by  │
  │    WA servers (no delivery, no error receipt).                       │
  │                                                                      │
  │  Fix applied (src/message-engine/interactive.js):                    │
  │    copy_code: String(textToCopy)                                     │
  │                                                                      │
  │  Regression tests added: Phase 22 WA-05 block (2 tests)             │
